O que são Conventional Commits?

Conventional Commits são convenções de nomenclatura para mensagens de commit que enfatizam com clareza as modificações realizadas. Essa abordagem padroniza as mensagens de commit, tornando-as mais informativas e fáceis de entender.

Eles são estruturados de forma a destacar o tipo de mudança feita, o escopo da mudança, e uma descrição clara da alteração ou adição.

Estrutura de uma Mensagem de Commit

tipo[escopo(opcional)]: descrição
---LINHA EM BRANCO---
[corpo (opcional)]
---LINHA EM BRANCO---
[rodapé (opcional)]

Exemplo completo de commit com todos os campos preenchidos: 

feat(api): adiciona endpoint para criação de usuários

- Adiciona um novo endpoint na API para criação de usuários
- Recurso é necessário para integrar com o módulo de autenticação

BREAKING CHANGE: altera a estrutura de resposta do endpoint de login
Agora a resposta inclui o campo "userId" para melhor integração

Refs: #123
Reviewed-by: Michael Douglas
Closes: #456

Este exemplo mostra todos os campos: tipo, escopo, descrição, corpo e rodapé. O campo BREAKING CHANGE destaca alterações que quebram compatibilidade com versões anteriores. Os campos Refs, Reviewed-by e Closes fornecem informações adicionais sobre a alteração.

Informações sobre cada parte da estrutura

Type
Deve ser um dos seguintes:

  • build: Alterações que afetam o sistema de build ou dependências externas (exemplos: gulp, broccoli, npm).
  • ci: Alterações em arquivos de configuração de integração contínua e scripts (exemplos: Travis, Circle, BrowserStack, SauceLabs).
  • docs: Apenas alterações na documentação.
  • feat: Uma nova funcionalidade.
  • fix: Uma correção de bug.
  • perf: Uma alteração de código que melhora o desempenho.
  • refactor: Alteração de código que não corrige um bug nem adiciona uma funcionalidade.
  • style: Alterações que não afetam o significado do código (espaços em branco, formatação, ponto e vírgulas ausentes, etc).
  • test: Adição ou correção de testes existentes.
  • Entre outros...

Scope
O escopo deve ser o nome da unidade relevante do projeto, como percebido por quem está lendo o changelog gerado a partir das mensagens de commit.

Subject
O assunto contém uma descrição sucinta da alteração:

  • Use o presente do indicativo e o modo imperativo: "alterar" e não "alterado" ou "altera".
  • Não inicie com letra maiúscula.
  • Não termine com ponto (.).

Body
Assim como no assunto, use o presente do indicativo e o modo imperativo: "alterar" e não "alterado" ou "altera". O corpo deve incluir a motivação para a alteração e contrastar com o comportamento anterior.

Footer
O rodapé deve conter qualquer informação sobre Breaking Changes e também é o local para referenciar problemas no GitHub que este commit resolve.

Breaking Changes devem começar com a palavra BREAKING CHANGE:, seguida por um espaço ou duas novas linhas. O restante da mensagem do commit deve ser usado para descrever a mudança.
